# 01 — 系统总体设计

## 1.1 系统概述

本系统为**计算机联锁模拟仿真系统**，采用纯软件技术在单台计算机上实现计算机联锁系统全部功能。系统将上位机操作平台、底层联锁逻辑和现场设备仿真三个层次集成为一体，模拟真实铁路车站计算机联锁（Computer Based Interlocking, CBI）系统。

### 1.1.1 模拟车站概况

模拟对象为一座**单线三股道中间站**（XX中间站），具备以下行车条件：

- 两端单线区间：上行端（JXG/JSG方向）和下行端
- 站内三条到发线：3G（上股）、IIG（正线中股）、1G（下股）
- 安全线（SFXG）一条，与1G之间通过5号道岔连接
- 具备接车、发车、通过及调车作业能力

### 1.1.2 设计目标

1. **完整性**：实现联锁系统上位机操作、联锁逻辑运算、现场设备仿真三层全部功能
2. **正确性**：联锁逻辑严格遵循"故障-安全"原则，确保行车安全
3. **可视化**：提供基于SVG的站场平面图，设备状态实时动态显示
4. **交互性**：支持点击站场图设备进行操作，提供面板式集中操作
5. **仿真性**：支持信号断丝、道岔故障、轨道占压等故障模拟

## 1.3 核心设计原则

### 1.3.1 进路驱动模型

系统采用**进路驱动**（Route-Driven）信号控制模型，即进站/出站信号机的显示**不由操作员直接控制**，而是由进路状态自动驱动：

```
操作员办理进路 → 联锁检查通过 → 道岔自动转换并锁闭 → 区段锁闭 → 信号自动开放
列车驶入信号机内方 → 信号自动复红
进路兑现/取消 → 道岔解锁、区段解锁、信号复红
```

这种设计与真实铁路联锁系统一致，体现了"信号是进路的表示"这一基本设计思想。

### 1.3.2 故障-安全原则

系统遵循铁路信号"故障-安全"（Fail-Safe）原则：

- **信号故障时强制红灯**：断丝故障模拟下信号机自动灭灯，进路拒绝开放该信号
- **道岔故障时自动封锁**：道岔故障后自动封锁，所有操作被拒绝
- **有车占用时禁止解锁**：进路路径上有列车时，拒绝解锁操作
- **进路锁闭时道岔禁动**：被进路锁闭的道岔拒绝单独操作

### 1.3.3 占用门控

系统实现了"红灯挡占用"的占用门控机制（即无开放进路的站内区段不能被占用）：

- **区间（JXG/JSG）**：允许自由占用（模拟区间来车）
- **站内区段**：必须属于一条已开放/占用中的进路才能被占用
- **红灯＝无开放进路＝不能占用**

### 1.3.4 数据持久化

所有运行状态通过 `localStorage` 自动保存，页面刷新后自动恢复：

- 设备状态（区段、道岔、信号机）
- 活动进路列表
- 列车列表
- 最近 100 条操作日志

## 1.4 运行流程

### 1.4.1 系统启动

```
index.html → main.js → 创建Vue应用 → 安装Pinia → 安装Vue Router → 安装Naive UI
  → 挂载#app → App.vue → router-view → Dashboard.vue
  → 初始化railway store（加载localStorage或默认状态）
  → 渲染站场图 + 顶部栏 + 侧栏 + 初始面板(进路面板)
```

### 1.4.2 操作响应流程

```
用户操作（点击按钮/站场图）
  → 面板组件调用 interlocking store 的对应方法
  → interlocking 执行联锁条件检查
  → 若通过：修改 railway store 中的数据 + 记录日志 + 保存localStorage
  → 若拒绝：返回错误原因 + 记录日志
  → railway store 响应式变化 → 所有组件自动更新（站场图/面板/顶部栏）
  → 通过 useNotify 弹出全局消息条
```
