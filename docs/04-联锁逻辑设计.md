# 04 — 联锁逻辑设计

## 4.1 联锁逻辑总览

联锁逻辑是系统的核心，全部实现在 `src/stores/interlocking.js` 中，以 **15个导出函数** 的形式提供操作接口。逻辑遵循铁路信号"故障-安全"原则，确保任何操作不会导致危险的系统状态。

### 4.1.1 函数分类

| 类别 | 函数 | 功能 |
|------|------|------|

## 4.2 进路办理流程

### 4.2.1 进路选排与锁闭 `establishRoute()`

```
输入: routeId（进路模板ID）

步骤1: 验证进路模板存在
  if 模板不存在 → 返回失败

步骤2: 敌对进路检查
  hasConflictingRoutes(template)
  → 检查与所有 activeRoutes 的路径区段交叉和道岔位置冲突
  if 有冲突进路 → 返回失败（含冲突进路名称）

步骤3: 区段空闲检查
  发车进路: 跳过始端股道（允许待发列车占用）
  接车/调车: 检查全部区段
  if 有区段占用/锁闭 → 返回失败

步骤4: 道岔可用检查及自动转换
  for 每个所需道岔:
    if 当前位置 != 所需位置:
      if 道岔 locked|singleLocked|blocked|defective → 返回失败
      否则自动转换至所需位置
      记录日志 "道岔自动转换：X号道岔 → 定位/反位"

步骤5: 锁闭轨道区段
  将进路路径所有区段 locked = true

步骤6: 锁闭道岔
  将进路所需所有道岔 locked = true

步骤7: 添加活动进路
  activeRoutes.push({...template, status:'locked', establishedAt:now})
  记录日志 "进路锁闭：进路名称 锁闭成功"

步骤8: 保存状态到 localStorage
```

### 4.2.2 信号开放 `openRoute()`

```
输入: routeId

步骤1: 验证进路存在且状态为 locked
步骤2: 检查信号机是否故障
步骤3: 根据进路类型设置灯色
  reception/departure → green（绿灯）
  shunting → yellow（黄灯）
步骤4: 进路状态置 opened，记录 openedAt 时间
步骤5: 记录日志 "信号开放"
步骤6: 保存状态
```

### 4.2.3 敌对进路检查 `hasConflictingRoutes()`

```javascript
// 检查新进路与所有活动进路是否存在冲突
function hasConflictingRoutes(newRoute):
  for each activeRoute in activeRoutes:
    // 冲突条件1: 路径区段有交叉（同一区段被两个进路使用）
    pathConflict = newRoute.path 与 activeRoute.path 有交集
    // 冲突条件2: 同一道岔要求不同位置
    turnoutConflict = 新进路要求某道岔位置与已激活进路要求相反

    if pathConflict OR turnoutConflict:
      return activeRoute   // 返回冲突的进路

  return null   // 无冲突
```

## 4.3 进路解锁流程

### 4.3.1 正常解锁 `unlockRoute()`

```
输入: routeId

步骤1: 验证进路存在
步骤2: 区段出清检查
  遍历进路路径所有区段
  if 任何区段仍占用 → 返回失败（附占用区段名称）
步骤3: 执行解锁 _doUnlock()
步骤4: 从活动进路列表删除
步骤5: 记录日志 "进路解锁"
步骤6: 保存状态
```

### 4.3.2 取消进路（含人工解锁）`cancelRoute()`

```
输入: routeId, force(是否强制)

步骤1: 验证进路存在
步骤2: 非强制模式
  if 任何区段占用 → 返回 {needForce:true}，提示需人工解锁
步骤3: 强制模式（force=true）
  跳过区段出清检查，直接解锁
步骤4: 执行解锁 _doUnlock()
步骤5: 删除活动进路
步骤6: 记录日志 "取消进路（人工强制取消）"
步骤7: 保存状态
```

### 4.3.3 进路兑现自动解锁 `autoReleaseFulfilledRoutes()`

此函数在每次轨道区段状态变化（占用/出清）时自动调用，实现**列车到达终点后进路自动释放**：

```
触发时机: 每次 occupySection() / clearSection() 之后自动调用

for each 活动进路(status='occupied'):
  dest = path[last]     // 终点区段
  earlier = path[0..last-1]  // 前段区段

  if dest被占用 AND 前段全部出清:
    _doUnlock(route)    // 释放道岔/区段锁 + 信号复红
    删除活动进路
    记录日志 "进路兑现，列车到达终点，进路自动解锁"
```

### 4.3.4 解锁执行 `_doUnlock()`

```
步骤1: 解锁轨道区段
  route.path 中每个区段 locked = false

步骤2: 解锁道岔（检查其他进路）
  for each 道岔:
    if 没有其他活动进路还在使用该道岔:
      该道岔 locked = false

步骤3: 关闭防护信号
  route.entrySignal 的信号机 aspect = 'red'
```

## 4.4 占用门控逻辑

### 4.4.1 占用区段 `occupySection()`

实现"红灯挡占用"机制，模拟真实联锁中的信号防护：

```
输入: sectionId

步骤1: 区段存在性检查
步骤2: 重复占用检查
  if 区段已占用 → 返回失败

步骤3: 门控检查（核心）
  sectionOccupiable(sectionId):
    区间(JXG/JSG): 允许自由占用（无需进路）
    站内区段: 检查是否有已开放/占用中的进路包含该区段
      if 无开放进路 → 返回失败 "无开放进路（信号红灯），不能占用"

步骤4: 设置区段 occupied = true
步骤5: 更新关联进路状态
  activeRoutes中 status='opened' 且 path含该区段 → status='occupied'
步骤6: 调用 autoReleaseFulfilledRoutes()
步骤7: 记录日志 "轨道占用"
步骤8: 保存状态
```

### 4.4.2 出清区段 `clearSection()`

```
输入: sectionId

步骤1: 区段存在性检查
步骤2: 设置区段 occupied = false

步骤3: 信号自动复红（核心安全逻辑）
  for each 活动进路(status=opened|occupied):
    if 该区段是进路的 path[0]（内方第一区段）:
      防护信号机 aspect = 'red'
      记录日志 "信号自动复红：列车驶过，XX信号自动复红"

步骤4: 调用 autoReleaseFulfilledRoutes()
步骤5: 记录日志 "轨道出清"
步骤6: 保存状态
```

### 4.4.3 自动复红触发条件

| 情形 | 触发条件 | 说明 |
|------|---------|------|
| 列车驶过 | 进路path[0]区段出清 | 列车驶离信号机内方第一区段 |
| 进路解锁 | _doUnlock调用 | 解锁时将防护信号复红 |
| 进路兑现 | 列车到达终点 | 自动解锁时复红 |

## 4.5 道岔操作逻辑

### 4.5.1 单独操作 `operateTurnout()`

```
操作约束（按优先级检查）:
  1. 道岔故障 → 拒绝："X号道岔故障，无法操作"
  2. 被进路锁闭 → 拒绝："X号道岔被进路锁闭，无法单独操作"
  3. 已单锁 → 拒绝："X号道岔已单独锁闭，请先解锁"
  4. 已封锁 → 拒绝："X号道岔已封锁，无法操作"
  5. 已在目标位置 → 拒绝："X号道岔已在定位/反位"

通过后: 设置道岔位置，记录日志，保存状态
```

### 4.5.2 单锁/封锁操作约束表

| 操作 | locked | singleLocked | blocked | defective | 是否可以 |
|------|--------|-------------|---------|-----------|---------|
| operateTurnout | ✗ | ✗ | ✗ | ✗ | ✅ |
| operateTurnout | ✓ | - | - | - | ❌ 进路锁禁动 |
| operateTurnout | - | ✓ | - | - | ❌ 单锁禁动 |
| operateTurnout | - | - | ✓ | - | ❌ 封锁禁动 |
| operateTurnout | - | - | - | ✓ | ❌ 故障禁动 |
| singleLock | ✓ | - | - | - | ❌ 进路锁优先 |
| singleLock | - | - | ✓ | - | ❌ 已封锁 |
| block | ✓ | - | - | - | ❌ 进路锁优先 |
| block | - | ✓ | - | - | ❌ 先解单锁 |

### 4.5.3 故障设置特殊逻辑 `setTurnoutDefective()`

```
设置故障: defective=true → 自动封锁 blocked=true
清除故障: defective=false → 自动解封 blocked=false
```

## 4.6 信号机操作逻辑

### 4.6.1 人工设置信号 `manualSetSignal()`

```
操作约束:
  1. 信号机故障 → 拒绝
  2. 信号机为 automated=true（进站/出站）→ 拒绝
     "X号信号由进路自动控制，请办理进路开放"
  3. 调车信号机 → 允许手动设置灯色

通过后: 设置信号 aspect，记录日志，保存状态
```

### 4.6.2 断丝故障 `setSignalDefective()`

```
设置故障: defective=true → aspect强制设为'red'
清除故障: defective=false → aspect保持'red'，等待进路重新开放
```

## 4.7 列车操作逻辑

### 4.7.1 添加列车 `addTrain()`

```
输入: number(车次), sectionId(初始区段), direction(方向)

步骤1: 区段存在性 + 占用检查
步骤2: 占用门控检查 (sectionOccupiable)
  → 区间自由，站内须有开放进路
步骤3: 生成列车ID (自增)
步骤4: 占用目标区段 (occupySection)
步骤5: 添加列车对象到 trains 数组
步骤6: 记录日志，保存状态
```

### 4.7.2 移动列车 `moveTrain()`

```
输入: trainId, targetSectionId, enforce(是否执行联锁检查)

步骤1: 列车存在性检查
步骤2: 目标区段存在性 + 占用检查
步骤3: 联锁检查 (enforce=true时)
  → sectionOccupiable(targetSectionId) 门控检查
步骤4: 出清当前区段 (clearSection，触发自动复红)
步骤5: 占用目标区段
步骤6: 更新关联进路状态
步骤7: 调用 autoReleaseFulfilledRoutes()
步骤8: 记录日志，保存状态
```

### 4.7.3 出清列车 `removeTrain()`

```
步骤1: 列车存在性检查
步骤2: 出清列车所在区段 (clearSection)
步骤3: 从 trains 数组删除
步骤4: 记录日志，保存状态
```

## 4.8 联锁条件汇总

| 操作 | 区段空闲 | 无敌对进路 | 道岔可用 | 信号正常 | 无车占用 | 其他 |
|------|---------|-----------|---------|---------|---------|------|
| 锁闭进路 | ✅ 接车全段/发车跳过始端 | ✅ | ✅ 自动转岔检查 | - | - | - |
| 开放信号 | - | - | - | ✅ 非故障 | - | 进路已锁闭 |
| 正常解锁 | - | - | - | - | ✅ 全部出清 | 进路存在 |
| 取消进路 | - | - | - | - | - | force=true可跳过 |
| 单操道岔 | - | - | ✅ 四重锁检查 | - | - | - |
| 添加列车 | - | - | - | - | ✅ 目标空闲 | 站内需进路 |
| 移动列车 | - | - | - | - | ✅ 目标空闲 | 站内需进路 |
