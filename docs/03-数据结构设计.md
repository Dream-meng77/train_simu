# 03 — 数据结构设计

## 3.1 总体数据架构

系统数据模型由 Pinia Store 管理，分为两个核心 Store：

```
railway.js（数据模型层）
  ├── stationLayout    ── 站场设备状态（响应式对象）
  │   ├── trackSections ── 轨道区段（8个）
  │   ├── turnouts      ── 道岔（5个）
  │   ├── signals       ── 信号机（10个）
  │   └── devices       ── 其他设备（3个）
  ├── routeTemplates   ── 进路模板（13条，静态）
  ├── activeRoutes     ── 活动进路（响应式数组）
  ├── trains           ── 列车列表（响应式数组）
  └── operationLog     ── 操作日志（ref数组）

interlocking.js（联锁逻辑层）
  └── 15个纯函数，读取/修改 railway store 中的数据
```

## 3.2 轨道区段数据模型

### 3.2.1 数据结构

```javascript
trackSections: {
  'JXG': {
    id: 'JXG',           // 区段编号（唯一标识）
    name: '进站区间',     // 中文名称
    occupied: false,     // 是否被列车占压（true=占用）
    locked: false,       // 是否被进路锁闭（true=锁闭）
    blocked: false       // 是否被封锁（true=不可用）
  },
  // ... 其余7个区段结构相同
}
```

### 3.2.2 状态转换表

| 当前状态 | 事件 | 新状态 | 条件 |
|---------|------|--------|------|
| 空闲 | 进路锁闭 | 锁闭 | 进路包含该区段 |
| 空闲 | 列车占压(区间) | 占用 | 区间允许自由占用 |
| 空闲 | 列车占压(站内) | 占用 | 须属于已开放进路 |
| 锁闭 | 列车占压 | 占用 | 须属于已开放进路 |
| 占用 | 列车出清 | 空闲 | 任何时刻允许 |
| 锁闭 | 进路解锁 | 空闲 | 区段未占用 |
| 占用 | 进路解锁 | 占用 | 区段仍有车 |
| 任意 | 封锁 | 封锁 | 手动设置 |
| 封锁 | 解除封锁 | 空闲 | 手动解除 |

## 3.3 道岔数据模型

### 3.3.1 数据结构

```javascript
turnouts: {
  '1': {
    id: '1',               // 道岔编号
    name: '1号道岔',        // 名称
    position: 'normal',    // 位置: 'normal'(定位) | 'reverse'(反位)
    locked: false,         // 进路锁状态
    singleLocked: false,   // 单锁状态
    blocked: false,        // 封锁状态
    defective: false       // 故障状态
  },
  // ... 其余4个道岔结构相同
}
```

### 3.3.2 道岔位置转换方向

| 道岔 | 定位(normal) | 反位(reverse) |
|------|-------------|---------------|
| 1# | 直通IIG | 侧通3G/1G方向 |
| 3# | IIG直行/3G接通 | 1G渡线接通 |
| 4# | IIG直通IIBG | 3G↔IIBG汇合 |
| 2# | IIG直通IIBG | 1G↔IIBG汇合 |
| 5# | 1G直行 | 通安全线(SFXG) |

### 3.3.3 锁定状态组合规则

道岔同时最多处于一种锁定状态（按优先级排序）：

```
defective=true  → 同时 blocked=true（故障自动封锁）
blocked=true    → 禁止所有操作
locked=true     → 被进路锁闭，禁止单操/单锁
singleLocked=true → 被单独锁闭，禁止单操
以上全为 false  → 空闲，可单操
```

## 3.4 信号机数据模型

### 3.4.1 数据结构

```javascript
signals: {
  'X': {
    id: 'X',               // 信号机编号
    name: 'X进站信号',      // 名称
    aspect: 'red',         // 灯色: 'green' | 'yellow' | 'red'
    type: 'entry',         // 类型: 'entry'(进站) | 'exit'(出站) | 'shunting'(调车)
    automated: true,       // 是否由进路自动控制
    defective: false       // 是否断丝故障
  },
  // ... 其余9个信号机结构相同
}
```

### 3.4.2 信号类型与控制方式

| 类型 | type值 | automated | 控制方式 |
|------|--------|-----------|---------|
| 进站信号 | 'entry' | true | 进路驱动，禁止手动 |
| 出站信号 | 'exit' | true | 进路驱动，禁止手动 |
| 调车信号 | 'shunting' | false | 可手动操作 |

### 3.4.3 信号状态转换

```
红灯(red) ──[进路开放]──→ 绿灯(green) [接车/发车] 或 黄灯(yellow) [调车]
绿灯/黄灯 ──[列车驶过内方]──→ 红灯(red)    [自动复红]
绿灯/黄灯 ──[进路解锁]────→ 红灯(red)      [自动复红]
任意 ──[断丝故障]──────→ 红灯(red), defective=true
断丝故障 ──[故障恢复]───→ 红灯(red), defective=false
```

## 3.5 进路数据模型

### 3.5.1 进路模板（静态定义）

```javascript
routeTemplates = [
  {
    id: 'X-IIG',                    // 进路编号
    name: 'X→IIG（接车进正线）',     // 进路名称
    type: 'reception',              // 类型: 'reception'|'departure'|'shunting'
    path: ['JXG', 'IIAG', 'IIG'],   // 路径区段序列（有序）
    entrySignal: 'X',               // 防护信号机编号
    turnouts: [                     // 所需道岔及位置
      { id: '1', position: 'normal' }
    ]
  },
  // ... 共13条模板
]
```

### 3.5.2 活动进路（运行时动态）

```javascript
activeRoutes = [
  {
    id: 'X-IIG',               // 对应模板id
    name: 'X→IIG（接车进正线）',
    type: 'reception',
    path: ['JXG', 'IIAG', 'IIG'],
    entrySignal: 'X',
    turnouts: [{ id: '1', position: 'normal' }],
    status: 'locked',           // 进路状态: 'locked'|'opened'|'occupied'
    establishedAt: '2024-...',  // 建立时间（ISO格式）
    openedAt: null,             // 开放时间（ISO格式）
    cancelCountdown: null       // 取消倒计时（秒）
  }
]
```

### 3.5.3 进路状态机

```
         ┌──[锁闭进路]──→ locked(锁闭)
         │                    │
         │              [开放信号]
         │                    ↓
         │               opened(开放)
         │                    │
         │              [列车占压]
         │                    ↓
         │              occupied(占用) ──[列车到达终点+前段出清]──→ 自动兑现解锁(移除)
         │                    │
         │              [取消进路/正常解锁]
         │                    │
         └────────────────────┴──→ 进路解锁(移除)
```

## 3.8 localStorage 持久化

### 3.8.1 存储键名

```
Key: 'railway_state'
```

### 3.8.2 存储内容

```javascript
{
  trackSections: { /* 8个区段完整状态 */ },
  turnouts:      { /* 5个道岔完整状态 */ },
  signals:       { /* 10个信号机完整状态 */ },
  activeRoutes:  [ /* 活动进路列表 */ ],
  trains:        [ /* 列车列表 */ ],
  nextTrainId: 5,                     // 下一列车ID
  operationLog: [ /* 最近100条日志 */ ]
}
```

### 3.8.3 加载合并策略

系统采用字段级合并策略，而非整体替换：

1. 读取 localStorage 中的 JSON 数据
2. 逐字段（trackSections / turnouts / signals）将保存的动态状态合并到代码中定义的默认结构中
3. 即只覆盖 `occupied`、`locked`、`position` 等**动态状态字段**
4. 保留代码中新增的字段（如 `automated`）不被覆盖

### 3.8.4 清理策略

- 系统全体复位时：调用 `localStorage.removeItem('railway_state')` 清除存储
- 日志条数控制：保存时截取最近 100 条
- 日志总量控制：内存中限制 500 条上限

## 3.9 数据流图

```
用户操作（UI组件）
      │
      ▼
interlocking.js 方法调用
      │
      ├──[联锁检查]──→ 失败 → 返回 {success:false, message} → UI 消息条
      │
      └──[检查通过]──→ 修改 railway.js 数据
            │               │
            │         ┌─────┴──────┐
            │         │            │
            │    addLog()    saveToStorage()
            │         │            │
            │    operationLog   localStorage
            │
            └──→ 响应式更新 ──→ UI组件自动重渲染
                  (站场图/面板/顶部栏/日志)
```
